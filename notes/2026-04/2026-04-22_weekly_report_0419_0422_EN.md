## TL;DR

This week we used a **semi-automatic agent ablation loop** (`run_chain.sh` driver + per-stage isolation + `summarize.py` aggregation) to tune GPT-OSS-20B MLPerf on 1×8 MI355X, and validated convergence end-to-end with `B11_full`:

- **End-to-end MLPerf wall time 11,916 → 9,963 s (−16.4 % vs base, −3.46 % vs last week's v2)**, val_loss 3.329 → **3.3247** ✅, margin 0.015, all 16 evals within Δ ≤ +0.025 of v2
- **Throughput 16.57 → 19.73 samples/s (+19.1 %)**, steady-state 988 → 776 ms/iter (−21.5 %), TFLOP/s/GPU ~418 → 533 (+27 %)
- **Samples-to-target stays 196,608** for both base and B11_full (same hit-iter band); eval window cut from ~58 s/run to ~29 s/run
- **Peak HBM ~98 % → ~83 %** (`grad_bf16` released 36 GB), leaving headroom for `mbs↑` / `use_turbo_attention`
- The same loop turned up two counter-intuitive findings on the mbs=4 exploration path (NUMA off ROI ×3, Triton RMSNorm ROI → 0), which directly rewrites next-stage ROI ranking

> mbs=4 short runs in §"Agent ablation loop": cumulative per-iter −18.8 %, TFLOPs 509 → 633 (+24 %), but per-sample convergence is ~30 % slower → estimated TTT +11 %, **not switching the main path yet**.

## 3. Agent semi-automatic ablation loop

This week we sketched the `pilot/agent/` tuning-agent skeleton (Orchestrator + Stage Worker + State-first protocol, see `pilot/README.v2.md`) and validated it with a minimal prototype (`/home/xiaompen/mlperf/ablations/M4_chain/`):

| Pilot role | this week's carrier |
|---|---|
| Orchestrator main loop | `run_chain.sh` driver (6 stages launched in order, fail-fast) |
| Stage Worker isolation | `run_one.sh` (own port + own log, no cross-stage context bleed) |
| Guardrail | `set -uo pipefail` + any non-zero stage → abort (only `--keep-going` overrides) |
| State / checkpoint | `logs/M4_C{0..5}.log` + `chain_summary.txt` |
| Settle aggregation | `summarize.py` produces a 6-row comparable table in one shot |

**Actual run**: launched under `nohup` and disconnected, **~2h 6min later 0 intervention / 0 OOM / 0 crash**, table below was produced directly:

| stage | added knob | mbs=4 step (ms) | mbs=4 cumulative Δ | mbs=4 marginal | mbs=2 marginal (04-17 same source) |
|---:|---|---:|---:|---:|---:|
| C0 | raw baseline | 1626.5 | — | — | — |
| C1 | + NUMA off | 1500.6 | −7.7 % | **−7.7 %** | −2.5 % |
| C2 | + `check_for_nan=false` | 1412.6 | −13.2 % | −5.9 % | −7.5 % |
| C3 | + `ddp_pad` | 1392.5 | −14.4 % | −1.4 % | −0.4 % |
| C4 | + `grad_reduce_in_bf16` | 1320.1 | −18.8 % | −5.2 % | −8.5 % |
| C5 | + `use_turbo_rms_norm` | **1319.9** | **−18.8 %** | **−0.02 %** | −1.5 % |

**Two counter-intuitive findings** (invisible if you only ran the full stack once):

1. **NUMA off scales up to −7.7 % under mbs=4** (only −2.5 % under mbs=2) — mbs=4 pushes HBM to 98.3 %, page balancing triggers frequent migrations; **any mbs=4 launcher must front-load `echo 0 > /proc/sys/kernel/numa_balancing`**.
2. **Triton RMSNorm suite ROI collapses to −0.02 % under mbs=4** (vs −1.5 % under mbs=2) — the bottleneck has moved to MoE permute / attention; **future mbs=4 work will not invest more eng. on RMSNorm**.
