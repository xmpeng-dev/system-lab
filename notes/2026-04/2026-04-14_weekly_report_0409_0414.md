# 周报: MoE CCO 优化 — MI355X (04/09 - 04/14)

**日期**: 2026-04-14

---

## 在做什么

MoE 层的 **CCO (Communication-Computation Overlap)** 优化，探索了两条路径：

**路径 1 — Multi-Stream Overlap（RCCL）**：将通信和计算分别放在不同 stream 上并行执行。实测发现 RCCL 受限于 collective 语义和 host-initiated 设计，多 stream 下通信与计算仍被串行化，无法真正 overlap。NVIDIA 已在 NCCL 2.28 中通过 Device API 和 Copy Engine Collectives 解决了这个问题，后续可关注 RCCL 是否借鉴跟进。

**路径 2 — Super Kernel 融合（当前重点）**：将 GEMM 计算与 All-to-All 通信融合在同一个 persistent kernel 内部，通过 HIP IPC 直接跨 GPU 读写内存，用 device-side flag 做 tile 级同步，从根本上规避 RCCL multi-stream 无法 overlap 的问题。目前正在开发原型。

```
路径 1 — Multi-Stream (受限于 RCCL):
  Stream 0: [Expert GEMM]
  Stream 1: [RCCL A2A]        ← 实际被序列化，无法并行

路径 2 — Super Kernel 融合 (当前方案):
  ┌──────────── 单个 Persistent Kernel ────────────┐
  │  Comm WG: IPC write tile 0 → tile 1 → ...      │
  │  Compute WG: GEMM tile 0 → tile 1 → ...        │  ← tile 级流水线
  │  两组 WG 通过 device-side flag 同步，无需回 host │
  └─────────────────────────────────────────────────┘
```

本周围绕这两条路径完成了四方面工作：基础 GEMM kernel、数据布局设计、overlap 方案调研与排障、端到端 baseline 建立。

---

## 主要成果

### 1. Hand-written Grouped GEMM 超越 CK (04/10)

Super kernel 内部的 expert GEMM 组件。在 DeepSeek-V3 规模 (M=128) 上大幅超越 CK：

| 配置 | 我们 | CK | 加速 |
|------|------|-----|------|
| FC1 (M128 K7168 N4096) | **497 TFLOPS** | 376T | +32% |
| FC2 (M128 K2048 N7168) | **498 TFLOPS** | 342T | +46% |

关键优化：3 阶段流水线 (issue→compute→commit)、`buffer_load` 硬件 prefetch、`ds_read` LDS 强制寻址。需 ROCm 7.2+。

### 2. Layout C — IPC Buffer 二维排序 + Pack 优化 (04/10)

Super kernel 的数据排布方案。发送端按 (dest_gpu, local_expert) 预排序，接收端 zero-gather 直接计算：

- 对比 5 种布局 (A-E)，**Layout C** 最优：XGMI 效率 ~95% + 无 gather 开销
- 256-stage sub-chunk signaling (8 src × 32 expert)
- Pack phase 8 项优化 (OPT-1~8)，理论 **~10× 加速**
- 预估端到端 **4.83ms (1.75× vs baseline 8.47ms)**

### 3. CCO Overlap 方案调研与排障 (04/09, 04/13, 04/14)

验证了三条在 MI355X 上实现 kernel 内通信的路径：

| 方案 | 状态 | 阻塞点 |
|------|:---:|------|
| RCCL multi-stream | ❌ | 架构限制：collective 语义 + host-initiated 设计，无法与 compute 并行 |
| rocSHMEM tile-level (Comet 移植) | ❌ | `hipErrorNoBinaryForGpu`：device bitcode 与 gfx950 不兼容 |
| HIP IPC + Triton system atomics | ❌ | 多轮 spin-wait hang：Triton gfx950 system-scope atomic codegen 不完整 |

**结论**：RCCL 从架构上不支持 overlap（NVIDIA 在 NCCL 2.28 通过 Device API / CE Collectives 解决，RCCL 未跟进）。需自行实现 kernel 内通信，最可行路径是 **HIP C++ IPC**（参考 DeepEP 的 `atomicAdd_system` + `ld_volatile_global`）。

### 4. MoE 端到端 Baseline (04/14)

量化了 overlap 的理论收益上限：

| 配置 | Baseline | Comm/Compute | 70% Overlap 加速 |
|------|----------|:---:|:---:|
| Small (2K tok) | 3.32ms | 85% | **1.29×** |
| DSV3 (mbs=2) | 17.15ms | 17% | **1.08×** |
| DSV3 (gbs=16) | 68.25ms | 24% | **1.13×** |

推理 (小 batch) 通信占比高，overlap 收益显著；训练 (大 batch) 计算主导，收益有限。
