# 周报: GPT-OSS-20B MLPerf 调优 — MI355X (04/19 - 04/22)

**日期**: 2026-04-22
**平台**: 1×8 MI355X, Primus + Megatron-LM, FP8 hybrid GEMM + bf16 weight
**主路径配置**: tp=pp=ep=1, mbs=2 / gbs=16
**汇总来源**: `/home/xiaompen/mlperf/benchmark_runs/PROGRESS_REPORT.md`

---

## TL;DR

本周采用一个**半自动 agent ablation loop**（`run_chain.sh` driver + per-stage isolation + `summarize.py` 汇总），在 1×8 MI355X 上对 GPT-OSS-20B MLPerf 做了一轮 tuning，并跑通 B11_full 全量收敛验证：

- **端到端 MLPerf wall time 11,916 → 9,963 s (−16.4 % vs base, −3.46 % vs 上周 v2)**, val_loss 3.329 → **3.3247** ✅, margin 0.015，16 个 eval 全部 Δ ≤ +0.025
- **吞吐 16.57 → 19.73 samples/s (+19.1 %)**，稳态 988 → 776 ms/iter (−21.5 %)，TFLOP/s/GPU ~418 → 533 (+27 %)
- **样本数 base 与 B11_full 同为 196,608**（同 hit iter 段）；eval 时段从 ~58 s/次 压到 ~29 s/次
- **显存峰值 ~98 % → ~83 %**（grad_bf16 释放 36 GB），为 mbs↑ / `use_turbo_attention` 留出空间
- 同条 ablation loop 在 mbs=4 探索路径上跑出两个反直觉发现（NUMA off ROI ×3、Triton RMSNorm ROI 归零），直接改写下一阶段 ROI 排序

> mbs=4 探索路径短跑见 §"Agent ablation loop"：单 iter 累积 −18.8 %, TFLOPs 509 → 633 (+24 %), 但单样本收敛慢 ~30 %, TTT 估 +11 %, **暂不切主路径**。

## 3. Agent 半自动 ablation loop

本周搭了 `pilot/agent/` 调优 agent 骨架（Orchestrator + Stage Worker + State-first protocol，详见 `pilot/README.v2.md`），先用最小原型 (`/home/xiaompen/mlperf/ablations/M4_chain/`) 跑通：

| Pilot 角色 | 本次承载物 |
|---|---|
| Orchestrator 主循环 | `run_chain.sh` driver（6 stage 顺序拉起, fail-fast） |
| Stage Worker 隔离 | `run_one.sh`（独立端口 + 独立 log, stage 间无 context 污染） |
| Guardrail | `set -uo pipefail` + 任意 stage 非 0 → abort（`--keep-going` 才放行） |
| State / checkpoint | `logs/M4_C{0..5}.log` + `chain_summary.txt` |
| Settle 汇总 | `summarize.py` 一次产出 6 行可对比表 |

**实际 run**：`nohup` 起来后断开，**~2h 6min 后 0 干预 / 0 OOM / 0 crash**, 直接拿到下面这张表：

| stage | 累加项 | mbs=4 step (ms) | mbs=4 累积 Δ | mbs=4 边际 | mbs=2 边际（04-17 同源）|
|---:|---|---:|---:|---:|---:|
| C0 | raw baseline | 1626.5 | — | — | — |
| C1 | + NUMA off | 1500.6 | −7.7 % | **−7.7 %** | −2.5 % |
| C2 | + `check_for_nan=false` | 1412.6 | −13.2 % | −5.9 % | −7.5 % |
| C3 | + `ddp_pad` | 1392.5 | −14.4 % | −1.4 % | −0.4 % |
| C4 | + `grad_reduce_in_bf16` | 1320.1 | −18.8 % | −5.2 % | −8.5 % |
| C5 | + `use_turbo_rms_norm` | **1319.9** | **−18.8 %** | **−0.02 %** | −1.5 % |

**两个反直觉发现**（如果只跑 full stack 一次根本看不到）：

1. **NUMA off 在 mbs=4 下放大到 −7.7 %**（mbs=2 下仅 −2.5 %）—— mbs=4 让显存压到 98.3 %，page balancing 触发频繁迁移；**任何 mbs=4 启动脚本必须前置 `echo 0 > /proc/sys/kernel/numa_balancing`**。
2. **Triton RMSNorm 套件在 mbs=4 下 ROI 归零 (−0.02 %)**（mbs=2 下 −1.5 %）—— 瓶颈已迁移到 MoE permute / attention；**未来 mbs=4 路径不再追加 RMSNorm 工程量**。
