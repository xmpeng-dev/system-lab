# 1024 卡训练稳定性总结（面试版）

## 1. 项目背景（一句话版本）

在 `32` 节点、`1024` 卡规模下做 DeepSeek-V3 风格 MoE 预训练，核心目标是：**先稳定、再提速、最后长跑**。

---

## 2. 我的职责（面试可直接说）

- 负责大规模训练启动脚本的并行拓扑与稳定性参数设计。
- 负责节点池筛选、网络参数固定、显存峰值治理与训练抖动排障。
- 负责把“能跑”升级到“可复现、可扩展、可长跑”。

---

## 3. 关键配置与设计逻辑（为什么这样配）

### 3.1 并行拓扑先行

- `NNODES=32`，总规模 1024 卡。
- 并行策略：`EP=8, PP=16, VPP=2`，即 `STAGE=32`。
- 为 `STAGE=32` 显式配置 `pipeline_model_parallel_layout`，避免自动切分带来的 stage 不均衡。

**经验**：1k 卡下先冻结拓扑（PP/EP/VPP），再调 batch；反过来通常会导致排障维度过多。

### 3.2 Batch 策略

- `MBS=2`，`GBS=128 * NNODES`（32 节点时为 4096）。
- 采用“节点线性扩展”的 GBS 初始策略，先保证收敛与稳定，再做吞吐优化。

**经验**：大规模起跑阶段用保守 MBS，减少首轮 OOM 与梯度抖动风险。

### 3.3 显存峰值治理（重计算）

- 使用 `RECOMP_IDS` 进行“分层打点”重计算，而非全层统一重算。
- 设计原则：前段保持密度保障内存安全，后段放锚点压峰值，同时控制额外算力损失。

**经验**：比 `recompute_num_layers` 粗粒度策略更容易在“稳定性”和“吞吐”之间找到平衡。

### 3.4 通信与网络固定

- 显式绑定网络接口：
  - `NCCL_SOCKET_IFNAME=ens9np0`
  - `GLOO_SOCKET_IFNAME=ens9np0`
- 显式指定 HCA：
  - `NCCL_IB_HCA=ionic_0:1,ionic_2:1,...`

**经验**：1k 卡场景不要依赖自动探测；接口/HCA 不固定是最常见的不稳定来源之一。

### 3.5 运行时稳定性开关

- `--numa` + `ENABLE_NUMA_BINDING=1`：提高 CPU/内存局部性，降低长尾抖动。
- `--manual_gc True --manual_gc_interval 1`：降低碎片导致的周期性卡顿。
- 默认关闭外部开销：
  - `--disable_wandb True`
  - `--disable_tensorboard True`
- 仅在窗口内 profile（step 6~7），避免全程 profiling 干扰主训练。

## 4. 稳定性方法论（面试官最看重）

我采用“三阶段推进法”：

1. **冒烟阶段（能跑）**
   - 小步数（如 10~100 iter）验证分布式初始化、通信、前后向链路完整性。
2. **稳态阶段（稳跑）**
   - 扩到目标节点数，观察 loss、step time 波动、OOM/timeout、节点尾延迟。
3. **长跑阶段（可交付）**
   - 拉长运行时长，验证无周期性抖动、无隐性内存爬升、无慢节点拖尾。

**核心原则**：每次只改一类变量（拓扑/通信/重计算/batch），保证问题可定位、结果可复现。

---

## 5. 常见故障与处理思路（高频问答素材）

### 5.1 通信超时或初始化卡住

- 现象：collective hang、部分 rank 不前进。
- 处理：
  - 固定 `NCCL_SOCKET_IFNAME` 与 `NCCL_IB_HCA`。
  - 缩小节点池，排除异常节点后逐步扩容。
  - 保持拓扑不变，仅替换节点，判断是“配置问题”还是“节点问题”。

### 5.2 OOM 或显存峰值波动大

- 现象：特定 stage 反复 OOM，或者偶发 OOM。
- 处理：
  - 优先调整 `RECOMP_IDS`，再考虑降低 MBS。
  - 保持 GBS 不变，避免收敛行为同时变化导致误判。
  - 对 pipeline layout 做均衡检查，避免 stage 热点。

### 5.3 吞吐不稳定（step time 抖动）

- 现象：吞吐均值还行，但方差大、P99 差。
- 处理：
  - 开启 NUMA 绑定，检查是否跨 NUMA 访问。
  - 开启手工 GC 观察是否降低周期性抖动。
  - 关闭非必要观测组件，先拿纯训练基线。

## 6. 我总结的可复用稳定性清单（Checklist）

- [ ] 并行拓扑（PP/EP/VPP）是否先冻结？
- [ ] pipeline layout 是否按 stage 显式配置？
- [ ] 网络接口与 HCA 是否显式绑定？
- [ ] 是否有“可用节点池”，并持续剔除异常节点？
- [ ] 是否先跑冒烟，再上稳态，再做长跑？
- [ ] 是否用分层重计算而不是全局粗放重算？
- [ ] 是否控制了监控/分析工具的额外开销？
- [ ] 是否做到“每次只改一类变量”？

---

## 7. 面试可讲的亮点（建议背诵）

- 我不是只调吞吐，而是先建立了“稳定性优先”的工程基线，保证 1k 卡训练可复现。
- 我把问题拆成四个可控维度：拓扑、通信、显存、节点池；每次单维度实验，缩短定位路径。
- 我在重计算上采用“分层打点”策略，实操上比全局重算更容易兼顾稳定和效率。
- 我把节点质量管理纳入训练流程，而不是把所有问题都归因于模型或框架配置。

## 8. STAR 版本（1 分钟讲完）

**S（情境）**：我们要在 1024 卡规模做 MoE 预训练，初期存在通信不稳、偶发 OOM 和吞吐抖动。  
**T（任务）**：目标是把任务从“偶尔能跑”变成“稳定可复现的长跑任务”。  
**A（行动）**：我先冻结 PP/EP/VPP 与 layout，再固定 NCCL/GLOO 接口与 HCA；引入分层重计算压峰值；建立节点池筛选机制；分冒烟/稳态/长跑三阶段验证。  
**R（结果）**：训练稳定性显著提升，排障效率提高，后续扩容和调优有了统一基线。  

> 面试时如果被追问“量化结果”，可补：稳定运行时长、OOM 次数下降、step time 方差下降、异常节点替换率等指标（按你实际数据填写）。

## 9. 风险与规范（面试加分项）

- 凭据（如 HF/WandB token）必须通过外部环境注入，禁止写入脚本默认值。
- 训练日志命名要包含节点、拓扑、批量参数，保证问题可追溯。
- 新参数上线前必须经过“小规模验证 -> 目标规模复验”双门禁。

## 10. 可替换模板（你后续只改这几项）

- 规模：`NNODES`、`MBS`、`GBS`
- 拓扑：`PRIMUS_EP`、`PRIMUS_PP`、`PRIMUS_VPP`、`pipeline_model_parallel_layout`
- 稳定性：`RECOMP_IDS`、NUMA、GC、通信接口/HCA
- 观测：`PROFILE` 开关与采样窗口

把这些参数模板化后，你的面试表达会更像“有方法论的 owner”，而不只是“调过一些参数的人”。
