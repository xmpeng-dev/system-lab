# plan — Plan Schema

## Purpose

`Plan` is the unit of execution: a fully-specified configuration that
`submit.run` can launch. This file defines its schema, conventions,
and how `env.diff` interacts with `cluster.env_baseline`.

Schema mirrors `pilot/README.md` §8.2.

## Schema

```yaml
plan_id: r3_p2                    # monotonic per session: r{round}_p{idx}
parent_baseline: r2_p1            # the plan this was derived from (PlanGraph parent)

parallelism:
  tp:   <int>                     # tensor parallel
  pp:   <int>                     # pipeline parallel
  dp:   <int>                     # data parallel
  ep:   <int>                     # expert parallel (MoE only)
  vpp:  <int>                     # virtual pipeline (interleaving) when pp > 1
  cp:   <int|null>                # context parallel; null if not used
  # Invariant: tp × pp × dp = world_size; ep usually divides dp or world_size
  #            (framework-specific — see constraints/config.md)

runtime:
  mbs:        <int>               # micro-batch size
  gbs:        <int>               # global batch size
  recompute:  none | selective | full
  recompute_pattern:              # optional, when recompute=selective
    kind: every-N | attn-only | mlp-only | expert-only
    n:    <int>                   # for every-N
  seq_len:    <int>
  dtype:      bf16 | fp16 | fp8

comm:
  bucket_size_mb: <int>
  overlap:        <bool>          # framework-level grad/comm overlap

env:                              # ONLY records deltas vs cluster.env_baseline
  baseline_ref: <ClusterProfile.env_baseline.version>
  diff:
    NCCL_MIN_NCHANNELS: 16
    NCCL_BUFFSIZE:       16777216
    PYTORCH_HIP_ALLOC_CONF: "expandable_segments:True,max_split_size_mb:512"

predicted:                         # Execution Model output, not measurement
  tps:           <float>
  mem_peak_gb:   <float>
  comm_ratio:    <float>
  bubble_ratio:  <float>
  confidence:    <float>           # 0..1

generated_by:                      # provenance, set by replan.md / projection.md
  bottleneck:   COMM_BOUND | ... | null     # null for initial plans
  strategy:     <path to optimization sub-skill that produced this candidate>
  axis_change:                     # the single axis that differs from parent_baseline
    key:    runtime.mbs
    from:   2
    to:     3
    type:   structural | cluster_shared | weakly_local | strongly_local
```

## env.diff semantics

- **Diff-only.** `env.diff` records ONLY the deviations from
  `cluster.env_baseline.<baseline_ref>`. Never duplicate baseline
  values here — that's how audit / replay / rollback stays sane.
- **Resolved env**. The actual env at execution time is
  `cluster.env_baseline ⊕ plan.env.diff` (later wins on conflict).
  The Snapshot's `resolved_env` field records the merged set.
- **Baseline upgrades.** When EnvSweep promotes a `cluster_shared`
  diff into the baseline, all open plans' diffs lose that key
  (it's now default). The plan's `env.baseline_ref` should be bumped
  to the new baseline version on next derivation.
- **Per-plan retention**. `weakly_local` and `strongly_local` env
  values stay in `Plan.env.diff` indefinitely; never auto-merged.
  See `axis_taxonomy.md`.

## axis_change provenance

Every non-root plan must record exactly one axis change vs parent
(the typical case). Multi-axis changes are allowed but discouraged:

- Single axis change → `axis_change` is one tuple.
- Multi axis change → `axis_change` is a list, AND `replan.md` should
  flag the candidate with `attribution: ambiguous` so Settle gives
  it less weight in promotion.

## ID conventions

Mirroring `state_machine.md` § Persistent identifiers:

```
r0_p_init_<n>     initial plans from PROJECTION
r0_p0             baseline plan (the initial champion)
r0_smoke          smoke variant of r0_p_init_*
r0_ref            tiny single-GPU reference (correctness)
r{N}_p{i}         challenger candidates in round N (i ≥ 1)
r{N}_envsweep_{i} env sweep candidates in round N
```

## Where Plans live in State

- **Initial plans** (PROJECTION output): `state/initial_plans.yaml`
  (a list).
- **Per-plan files** (one per candidate, generated by Re-Plan):
  `state/plans/<plan_id>.yaml`.
- **PlanGraph nodes** carry `plan_ref` pointing at the YAML file.

## Validation

Every Plan, before submit, MUST pass:

```python
constraint.check(plan, cluster_profile)         # parallelism legality
constraint.estimate_mem(plan)                   # OOM bound
constraint.check_env(plan.env.diff, baseline)   # env compatibility
```

Rejections feed back into Re-Plan; never reach `submit.run`.

## Scale-aware notes

The schema is identical across scales. Some keys are typically `null`
or `1` on single node:

- `parallelism.dp = 1` (typical), `parallelism.pp = 1` (typical),
  `parallelism.cp = null`.
- `inter_node_*` env keys (`NCCL_IB_*`, `NCCL_SOCKET_IFNAME`) won't
  appear in baseline or diff.
- For multi-node, `parallelism.dp` is non-trivial and `gbs = mbs × dp
  × accumulation` matters for correctness.
