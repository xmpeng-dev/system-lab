MI355X vs B200 — same EP=8 GBS=16 config

single node · 8 GPUs · matched config · ProfilerStep #17 MI355X: TP1 PP1 EP8 · GBS 16 · MBS 2 B200: TP1 PP1 EP8 · GBS 16 · MBS 2 apples-to-apples
Headline — apples-to-apples MI355X loses on this config

With MI355X forced into the same EP=8 GBS=16 layout the B200 uses, per-sample B200 delivers 60.4 ms/sample vs MI355X 92.3 ms/sample — a 1.53× advantage for B200. The structural cause is the MoE expert all-to-all: RCCL all-to-all on MI355X consumes 641 ms (43% of step) vs NCCL sendrecv on B200 consumes 284 ms (29%). And on MI355X the all-to-all runs on the compute stream so only 3.5% of NCCL hides behind compute, vs 20% on B200. This is exactly why the AMD-tuned config picks EP=1 (skip the a2a, use 288 GB HBM to keep every expert local) — see the earlier canvas for that comparison.

What the timelines tell us

B200 — communication on the critical path

ncclDevKernel_SendRecv runs on the same stream as the forward / backward GEMMs (stream 7), so MoE dispatch + combine serializes with compute and consumes 284 ms / 967 ms (29%).

Two extra NCCL streams (51, 55) handle DDP reduce-scatter and all-gather, hiding 20% of total NCCL behind compute.

Net: compute stream is 43% busy, mostly idle inside a2a windows. Lever: pull a2a off the compute stream onto a dedicated NCCL stream and overlap it with the next micro-batch's attention.

MI355X — RCCL all-to-all dominates the step

A single RCCL kernel ncclDevKernel_Generic_1 takes 933 ms = 63% of the 1477 ms step and runs on stream 0 alongside the dense compute. AllGather (271 ms) lands on the same stream; only 3.5% of total NCCL overlaps anything.

Four side streams (15–18) carry grouped-MoE GEMMs (~350 ms summed) but they only fire after the long a2a tail, so they add throughput without absorbing the bubble.

Net: compute stream is 36% busy and there is a hard ~37 ms idle window mid-step. RCCL itself is ~2.25× heavier than the NCCL equivalent on the same dispatch pattern, so even if the schedule were perfect MI355X would still trail B200 on this config — and the schedule is far from perfect.

Why the AMD-tuned config picks EP=1

On 288 GB HBM all 32 experts fit on each MI355X, so the optimal config for AMD is EP=1 — skip MoE all-to-all entirely and rely on DDP collectives that overlap cleanly. In that mode the same MI355X node hits 1129 ms / 32 samples = 35.3 ms/sample, beating B200's 60.4 ms/sample by 1.71×. This canvas isolates what happens when the config is forced to match B200 instead.

Methodology

Both traces produced by torch.profiler with record_shapes=false, with_stack=false; one active step per rank. Numbers come from rank 2 of each run on the same ProfilerStep #17. Opaque RCCL kernels are re-categorized via SPLIT_NCCL_BY_CPU in full_breakdown.py — the MoE dispatch all-to-all surfaces as ncclDevKernel_Generic_1 and is attributed to the a2a bucket here.
